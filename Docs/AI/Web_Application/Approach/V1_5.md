# V1.5 ‚Äî Vertical Slice: SVG Canvas, Drag, and JSON Out

## Goal

Build on V1.1's `EditorCore` + schema-driven panel by adding the **two interaction surfaces and one IO surface** that V1.1 deliberately deferred: a visible SVG canvas of the scene, pointer-drag to move entities, and a working `sceneToJSON` serializer with a "Show JSON" toolbar button.

By the end of V1.5, you should be able to:

1. Click "Add Entity" ‚Äî a rectangle appears in the canvas at the transform's default position.
2. Click the rectangle in the canvas ‚Äî the entity becomes selected; both `EntityList` and `PropertyPanel` update.
3. Drag the rectangle ‚Äî `x`/`y` update live in `EditorCore`, the panel inputs reflect the new values, and the `EntityList` selection follows.
4. Edit `x` in the panel ‚Äî the rectangle moves on the canvas (round-trip from panel back to canvas).
5. Click "Show JSON" ‚Äî a dialog/textarea appears containing the serialized scene, matching `Engine.md ¬ß6` shape (minus the components V1.5 doesn't have yet).

In-memory only. No file save/load, no asset upload, no other components, no JSON import.

---

## Why this slice (and what was cut from V2)

V1.1 paid for the load-bearing architecture: `EditorCore` ‚Üî React subscription, schema-driven UI, hexagonal boundary. V1.5 exercises **three smaller-but-real questions** that only show up once a canvas exists:

- **The Core ‚Üî canvas round-trip closes.** Drag mutates `x`/`y` via `core.setField`; the panel re-renders from the same snapshot. If this works for one entity, it works for all of them ‚Äî there is no "drag-specific" state path to maintain.
- **Coordinate translation lives in one place.** SVG `viewBox` + `getScreenCTM().inverse()` is the canonical way to translate pointer client coords to canvas coords. Wrapping it in a `screenToCanvas(svg, clientX, clientY)` helper means no component does the math twice; the next interaction (resize handles, marquee select, ‚Ä¶) reuses it untouched.
- **`sceneToJSON` is real, not stubbed.** V1.1 left `transform.serialize` throwing. V1.5 makes it return the exact `{ x, y, width, height }` shape from `Engine.md ¬ß6` and asserts it in a unit test. That test is the first concrete pin against drift between the editor and `magic_bridge.py`.

V1.5 **defers** to V2:

- Other components (`sprite`, `collider`, `physics`, `audio`, `player_controlled`) and field kinds (`bool`, `enum`, `colour-abgr`, `asset-ref`).
- `jsonToScene` deserializer + File System Access load/save. JSON is read-only output in V1.5; round-tripping is a separate slice.
- `AssetStore` (IndexedDB), texture rendering inside rects, audio preview.
- Validation UI, drag-to-reorder entity list, display-name editing, undo/redo.
- Resize handles, snap-to-grid, zoom/pan. Canvas stays fixed 1:1 at 480 √ó 272.

## Build order

Each step keeps `npm run typecheck` + `npm run lint` + `npm run test` green. The order is chosen so each step is independently demoable.

1. **Extend ESLint with the `io/` boundary.** Add a third block to `eslint.config.js` mirroring the `core/` and `schemas/` ones: `files: ['src/io/**/*.{ts,tsx}']`, `no-restricted-imports` forbidding `**/ui/**`. Land this *before* writing `io/serialization/sceneToJSON.ts` so the rule catches a wrong import on the very first commit, not retroactively.
2. **Extract `schemas/registry.ts`.** Move the `REGISTRY` object out of `EditorCore.ts` into its own file (typed `Record<string, ComponentSchema<unknown>>`). `EditorCore` imports it. No behaviour change; tests stay green.
3. **Implement `transform.serialize` + `deserialize` + round-trip test.** Replace the V1.1 stubs. `serialize` returns a plain object `{ x, y, width, height }`. `deserialize` validates the shape (typeof number on each field) and returns a `Transform`. Round-trip test: `deserialize(serialize(t))` deep-equals `t` for a few representative values.
4. **`io/serialization/sceneToJSON.ts` + test.** Declares `SceneJSON` / `EntityJSON` at the top. Pure function `sceneToJSON(scene: Scene): SceneJSON`. Iterates entities; for each, looks up each component's schema in the registry, calls `schema.serialize(data)`, places the result under `components[key]`. Output shape: `{ entities: [{ id, components: { transform: {...} } }] }`. Note: `noUncheckedIndexedAccess` is on, so registry lookups by string return `ComponentSchema<unknown> | undefined` ‚Äî handle the undefined case (skip + warn, or throw ‚Äî V1.5 throws; an unknown component key in the scene is a Core invariant violation, not user input). Test: a scene with one entity serializes to the exact object literal from Engine.md ¬ß6 (transform fields only; no `display_name`, no `player_controlled` until those features land).
5. **`ui/canvas/screenToCanvas.ts` + test.** `(svg: SVGSVGElement, clientX: number, clientY: number) => { x: number; y: number }`. Implementation uses `svg.getScreenCTM()` applied to a `DOMPoint`, then `.inverse()`. Throws `Error('SVG has no CTM')` if `getScreenCTM()` returns `null` ‚Äî in practice this only happens if the SVG is detached from the document, which a real pointer event guarantees against. Test in jsdom by stubbing `getScreenCTM` to return a known `DOMMatrix` and asserting the inverse-mapped point.
6. **`ui/Canvas.tsx` ‚Äî read-only first.** `useScene()`, render `<svg viewBox="0 0 480 272">` with one `<rect>` per entity that has a transform (`if (!entity.components.transform) return null` inside the map ‚Äî dead branch in V1.5 but cheap insurance for V2 audio-only entities). Selected entity's rect gets `aria-selected="true"`; styling lives in `index.css` (`rect[aria-selected="true"]`), mirroring the EntityList convention. Click on a rect calls `core.selectEntity(id)`. Background click (on the SVG itself, not a rect) is **not** wired in V1.5 ‚Äî see Open questions. Wire `<Canvas>` into `App.tsx` between the list and the panel; manually verify that adding entities populates the canvas and clicking a rect selects.
7. **Drag on `<rect>`.** Add `onPointerDown` to each rect:
   - Call `core.selectEntity(id)` *first* ‚Äî a click is a zero-distance drag, and selection should update even if the user never moves the pointer.
   - `e.currentTarget.setPointerCapture(e.pointerId)` ‚Äî guarantees subsequent move/up events go to this rect even if the pointer leaves its bounds.
   - Record drag start in component state: pointer canvas coords (via `screenToCanvas`) and the entity's current `(x, y)`.
   - `onPointerMove` (only when drag-active): compute new pointer canvas coords, derive `dx`/`dy` from start, call `core.setField(id, 'transform', 'x', startX + dx)` and same for `y`. `x`/`y` are float-kind in the schema, so non-integer drag positions are correct.
   - `onPointerUp` / `onPointerCancel`: release capture (`releasePointerCapture`), clear drag state.

   Manual verify: drag a rect, see panel inputs update live; edit panel inputs, see rect move.
8. **`ui/JsonView.tsx` + "Show JSON" button.** Component takes `{ open: boolean; onClose: () => void; json: string }`. Internally: `useRef<HTMLDialogElement>(null)` plus a `useEffect([open])` that calls `ref.current?.showModal()` when `open` flips true and `ref.current?.close()` when false. Add an `onClose` handler on `<dialog>` itself (fires on ESC and backdrop click) to call the prop. Body is a read-only `<textarea>` with `JSON.stringify(sceneToJSON(scene), null, 2)`. Toolbar button in `App.tsx` toggles a local `useState(false)`. The dialog content is computed from the live snapshot at render time ‚Äî if the user opens it, drags an entity, the dialog stays stale until reopened. (V1.5 acceptance; V2 can subscribe if it matters.)

Steps 1-5 are pure logic / config and have unit tests (or are config). Steps 6-8 are UI-only and rely on the manual checks below.

  - Click the rect ‚Üí it gains `aria-selected="true"` (visible stroke); list row "#1" highlights; panel shows the four inputs.
  - Drag the rect 50 px right and 30 px down ‚Üí panel `X` shows ~`50`, `Y` shows ~`30` updating live; rect stays under the pointer; release leaves it in place.

If drag drifts (rect doesn't stay under the pointer) or jumps on first move, the bug is in `screenToCanvas` or the start-coord capture ‚Äî fix before declaring V1.5 done.

## The two patterns you're learning this slice

- **`getScreenCTM().inverse()` for SVG coordinate translation.** Browsers maintain a current transformation matrix that maps the SVG's user-space (`viewBox` units) to screen pixels. To go the other way (pointer client coords ‚Üí canvas coords), invert it and apply to a `DOMPoint`. This handles `viewBox`, `preserveAspectRatio`, page scroll, and CSS scaling for free ‚Äî doing the math by hand is the classic "works on my monitor, broken on retina" trap.
- **`setPointerCapture` for drag.** Without it, a fast drag that briefly leaves the rect's bounding box stops getting `pointermove` events and the rect stalls mid-drag. Capture pins the event stream to the originating element until you release it. This is the boring-but-correct way to do pointer drag in 2026 ‚Äî no global `mousemove` listeners on `window`.

Both are platform features, not React patterns. The React side stays trivial: `onPointerDown`/`Move`/`Up` handlers that call `core.setField`. The Core class doesn't change at all.

## Open questions to resolve during V1.5

- **Should `display_name` appear in JSON output?** Engine.md ¬ß6 shows it on every entity, and notes it's "for the editor's bookkeeping." V1.1 deferred display-name *editing* to "Later". Decision for V1.5: **omit `display_name` entirely** until the editing UI lands. Adding the field later means one line in `sceneToJSON` and updating one snapshot test ‚Äî cheaper than maintaining a fake `"#1"` placeholder that round-trips into the binary as garbage.
- **Float precision in JSON output.** Drag produces floats with many decimals (e.g. `49.99999`). For V1.5, write them as-is; `JSON.stringify` handles it and the engine reads them as `float`. If output legibility becomes annoying, round to 3 decimals at serialize time ‚Äî but that's a polish call for V2, not an architectural decision.
- **Drag and selection ordering.** On `pointerdown`, call `selectEntity` *before* recording drag-start coords, so the panel updates immediately even if the user just clicks (no drag). Confirmed up-front: a click is a zero-distance drag; the same handler covers both.
- **What if an entity has no transform?** It still appears in `EntityList` (via id) but is skipped by `Canvas.tsx` (no rect to draw). V1.5 only ever creates `['transform']` entities, so this branch is dead code today ‚Äî but the canvas should `filter` rather than crash on the day V2 adds audio-only entities. One-line `if (!entity.components.transform) return null` inside the map.
- **Dialog vs inline panel for JSON.** `<dialog>` is one element, ships with backdrop + ESC-to-close, and doesn't fight the existing layout. Picked over a slide-out panel which would need its own state in `App.tsx` and CSS. Revisit only if "Show JSON" needs to stay open while editing (it doesn't in V1.5; importing JSON is V2 work).
- **Background-click deselect on the canvas.** Skipped in V1.5. Adding it means an `onClick` on the `<svg>` itself with an `if (e.target === e.currentTarget) core.selectEntity(null)` guard so rect clicks don't bubble-deselect. Trivial to add but it's a UX call (some editors deselect, some don't), and V1.5 doesn't need a deselect path to demo any of its acceptance criteria. Revisit when validation UI lands and "no selection" actually means something distinct from "selection unchanged."
- **Selection-on-drag-start ordering.** `selectEntity` is called at the top of `onPointerDown`, before drag-start coords are recorded. Reasoning: a click is a zero-distance drag; the same handler must cover both, and selection should be visible the instant the pointer goes down ‚Äî not on `pointerup` which would feel laggy on long drags. Decided.
