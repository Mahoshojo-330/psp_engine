# V2.4 — Vertical Slice: JSON Import, File Save/Load, Entity Reorder, Display Name

## Goal

Close the V2 loop. After V2.4:

1. **Save** a `scene.json` to the user's filesystem via the File System Access API (or an `<a download>` fallback in browsers without it).
2. **Open** a `scene.json` from disk, parse it through a real `jsonToScene` deserializer, and the editor reconstructs an identical scene (entities, components, draw order, display names).
3. **Drag-to-reorder** the entity list. Lower index = drawn first (Engine.md §11). Reorder is one undo step.
4. **Display names** are editable inline in the entity list and serialized at the top level of each entity's JSON.
5. The full V2 acceptance test in `Requirements_V2.md` passes end-to-end.

---

## Why this slice last

V2.4 is the round-trip slice. Every prior slice's contract gets tested transitively: import → run validator → display in panel → re-export → assert byte-equal-ish to original.

Order rationale (restated from `Requirements_V2.md`):

- Without V2.0, every reorder/displayname/import step would be a per-mutation history retrofit.
- Without V2.1, `jsonToScene` has no schemas to deserialize against beyond `transform`.
- Without V2.2, an imported `scene.json` referencing texture ID `5` has nothing to bind to.
- Without V2.3, an imported scene that violates Engine.md §13 silently loads instead of lighting up errors.

So V2.4 is the slice that has the most leverage per line of code — the prior four slices did the architectural work; V2.4 is mostly wiring and one new pure function (`jsonToScene`).

## Build order

1. **Add `displayName` and the `setDisplayName` mutation.** Update `Entity` type. `setDisplayName(id, name)` mutates inside a transaction; if `name === ''` set `displayName: undefined` (no empty string). Test the round-trip: setDisplayName then undo → undefined again.

2. **Add `moveEntity(id, toIndex)`.** Splice and re-emit. Transactional — one history entry per drop. Selection follows: if the moved entity was selected, it stays selected (selection is by id, not index).

   Test: move entity #2 to index 0; entities order is [#2, #1, #3]; selection unchanged; undo restores order.

3. **Update `sceneToJSON`** to include `display_name` at the top level when set. Test the Engine.md §6 fixture: Hero gets `display_name: "Hero"`, top-level.

4. **Implement `jsonToScene`.** Step-by-step parse:
   - Assert top-level `entities` is an array.
   - For each entity:
     - Read `id` (number, required).
     - Read optional `display_name` (string, optional). Stored as `displayName` on the Entity.
     - Read `components` (object, required).
     - Build the `components` record: for each key, look up schema in `REGISTRY`, call `schema.deserialize(value)`. Throw on unknown keys with a clear message naming the key and entity id.
     - Read top-level booleans against `REGISTRY` `isFlag` keys: if `entity.player_controlled === true`, add an empty-body `components.player_controlled = {}`. (Generalized: iterate `Object.values(REGISTRY).filter(s => s.isFlag)` — supports future flag components without changing this code.)
     - Errors throw with structured info: `{ entityId, componentKey?, fieldName?, message }`.
   - Reset selection to null. Reset id counter to `max(entity.id) + 1` so future `addEntity` doesn't collide.

   Tests:
   - Round-trip a fixture: `sceneToJSON(jsonToScene(input)) === input` (deep equal).
   - Reject malformed JSON (non-object, missing entities, unknown component, wrong field type) with messages naming the offender.
   - Trip Engine.md §6's exact example through `jsonToScene` → resulting `Scene` has 3 entities with the expected components and flags.

5. **Add `EditorCore.replaceScene(scene)`.** Pushes the current scene to history (so import is undoable), clears redo, replaces. Emits.

   Test: edit, then `replaceScene(otherScene)`, then undo → original scene restored. Redo → otherScene back.

6. **`IOAdapter` interface + `FileSystemAccessIOAdapter`.**
   - `saveScene(json, suggestedName)`: tries `window.showSaveFilePicker({ suggestedName, types: [{ description: 'PSP Engine scene', accept: { 'application/json': ['.json'] } }] })`. Fallback: synthesize an `<a>` with `download={suggestedName}` and `href={URL.createObjectURL(blob)}`, click programmatically, revoke.
   - `loadScene()`: tries `showOpenFilePicker(...)`. Fallback: synthesize a hidden `<input type="file" accept=".json">`, click programmatically, await `change`. Returns `{ filename, json: string }`. User cancel → resolves to `null` (the picker rejection in the modern API; the lack-of-change in the fallback). Don't conflate cancel with error.

   Feature detection at construction:

   ```ts
   const hasNativeFs = 'showSaveFilePicker' in window
   ```

   Tests in jsdom: `hasNativeFs === false`, fallback path is selected. Modern API path is exercised by manual checks (jsdom can't fake `showSaveFilePicker` reliably, and the test would just be a stub assertion).

7. **`<App>` toolbar wiring.** Replace "Add Entity" preset dropdown stays. Replace "Show JSON" with two buttons: **Save** (disabled when validation has errors) and **Open**. Save calls `ioAdapter.saveScene(json, suggestedName='scene.json')`. Open calls `ioAdapter.loadScene()`, parses with `jsonToScene`, calls `core.replaceScene`, then sets the import banner state.

   Show JSON: keep as a third toolbar button labelled "View JSON" so the inspect path V1.5 introduced doesn't disappear. Useful for debugging exports without round-tripping disk.

8. **`<ImportBanner>`.** Transient banner above the toolbar showing `Loaded scene.json — 3 entities, 0 errors`. Auto-dismiss after 5s, manually dismissable via close button. Uses `useValidation()` to count errors.

9. **EntityList: inline displayName editing.**
   - Row label is `displayName ?? '#{entity.id}'`.
   - Double-click → switch to `<input>` with the displayName as value (or empty string if unset). Focus → `core.beginTransaction()`. Onchange writes through `core.setDisplayName(id, value || '')`. Blur or Enter → `core.commitTransaction()`. Esc → `core.abortTransaction()`, switch back to label.
   - Reuse the focused-text-state pattern from V1.5's PropertyPanel input (text state + focused ref).

10. **EntityList: drag-reorder.**
    - Each `<li>` gains `onPointerDown`. Capture pointer; record start position and the entity index. On `pointerMove`, compute the would-be drop index from pointer Y vs the bounding rects of siblings (or use simpler logic: count how many rows above the pointer's current Y). Render a `data-drop-target` attribute on the row above which the drop would land; CSS shows a 2px line above it.
    - On `pointerUp`, if drop index differs from start index, call `core.moveEntity(id, dropIndex)` once. Single history entry.
    - On `pointerCancel`, abort drag (no mutation; no transaction was opened — moveEntity is the entire transaction-bearing call).

    Test the math without the DOM: extract a pure `computeDropIndex(rectTops, pointerY)` helper, unit test it. The DOM test uses `fireEvent.pointerDown/Move/Up` from `@testing-library/react` and asserts `core.moveEntity` was called with expected args.

11. **`main.tsx` composition.** Instantiate `FileSystemAccessIOAdapter`, pass to `<App>`. No async init needed (feature detection is sync); the AssetStore's async init from V2.2 is the only `await` in `main.tsx`.

Steps 1–6 are pure logic with unit tests. Steps 7–11 are UI and rely on manual checks plus the cumulative end-to-end test.

## Acceptance criteria

This is the V2 final acceptance test, restated from `Requirements_V2.md` §Verification with the per-slice unit tests assumed-green.

- `npm run typecheck && npm run lint && npm run test` — clean.
- `npm run dev` end-to-end:
  - **Build the Engine.md §6 scene from scratch.**
    - "Add Entity → Dynamic" → name it "Hero" via double-click + Enter. Add a `sprite` component. Upload `hero.png` (32×32). Set sprite's texture to ID 0. Toggle `player_controlled`.
    - "Add Entity → Static Solid" → name it "Floor". Position `(0, 250)` size `480×22`. Upload `floor.png`. Set sprite's texture to ID 1.
    - "Add Entity → Empty" → name it "Background Music". Add an `audio` component. Upload a 16-bit PCM WAV. Set `sound_id = 0`, `loop = true`, `volume = 0.7`.
    - Toolbar status reads "Valid."
  - **Reorder.** Drag Floor in the list to be above Hero. Canvas redraws Hero on top. Toolbar still "Valid."
  - **Save.** Click Save, pick a path, write `scene.json`. Open the file in a text editor; it matches the Engine.md §6 example shape: top-level `entities` array, `id`, `display_name`, optional `player_controlled`, `components` object, draw order [Floor, Hero, Background Music].
  - **Reload page.** AssetBrowser shows the three uploaded assets (IndexedDB persistence). Toolbar shows "Valid" against an empty scene (V2.4 doesn't auto-load; user opens manually).
  - **Open the saved file.** Click Open, pick the just-saved JSON. ImportBanner appears: "Loaded scene.json — 3 entities, 0 errors." EntityList shows Floor, Hero, Background Music in the saved order. Selecting Hero shows the same component values as before save. Toolbar reads "Valid."
  - **Re-save.** Click Save. Diff against the earlier file is empty (modulo the float-precision V1.5 open question; if it surfaces, we'd land a `roundTo3Dp` patch in V2.4 — see Open questions).
  - **Cmd+Z after Open** — scene reverts to the pre-import state (empty editor). Cmd+Shift+Z re-applies the import.
- **Pipeline confirmation.**
  - `python3 Pipeline/magic_bridge.py path/to/scene.json` → emits `scene.bin` without errors.
  - Convert `hero.png`, `floor.png` via `texture_converter.py` to produce `tex_0.raw`, `tex_1.raw`. Convert the WAV via `audio_converter.py` to `sfx_0.raw`.
  - Drop into `ms0:/PSP/GAME/PSP_Engine/scenes/` (or PPSSPP). Engine boots, Hero responds to D-pad, Floor catches Hero on fall, music loops.
- **Failure modes.**
  - Open a malformed `scene.json` (random text) → ImportBanner shows the parse error; previous scene unchanged. No half-applied state.
  - Open a `scene.json` whose schema-level fields are wrong type (e.g. `transform.x` is a string) → ImportBanner reports the offender (`entity 1, transform.x: expected number`); previous scene unchanged.
  - Open a `scene.json` that violates Engine.md §13 (e.g. sprite without transform) → ImportBanner says "Loaded with N errors"; scene loads but Save is disabled until fixed. Cmd+Z reverts to prior state if user wants to back out.

## Open questions to resolve during V2.4

- **Should opening a malformed file be an error toast or a banner?** Decision: **banner** (same lane as success). Toasts compete with the toolbar status badge. The banner has a close button and includes the parse error text; that's enough.
- **Should Save fail when there are errors, or save anyway?** Decision: **save is disabled when errors exist.** The user can View JSON to copy out manually if they really need to. The reasoning matches V2.3 — Engine.md §13 violations crash or render-blank, and "I built a game and it doesn't work" is exactly the failure mode the editor was supposed to prevent.
- **`replaceScene` and history.** Decision: **import is undoable** (push prior scene to history, clear redo). The user accidentally opening the wrong file shouldn't lose their prior work. Memory: same constraint as the rest of the history cap.
- **Float precision in JSON.** V1.5 open question revisited. Drag math produces values like `49.999999`. Decision for V2.4: **leave as-is**. JSON.stringify will print them; engine reads as float. Round-trip-equality via deep-equal still passes because the floats round-trip identically. If the JSON output legibility ever bites, a one-line `roundTo(value, 4)` in `transform.serialize` is the fix; not V2.4's call to make.
- **EntityList: drag handle vs whole-row drag.** Decision: **whole-row drag** (entire `<li>` is the handle). Adds `cursor: grab` to communicate it. A dedicated handle (≡ icon) is more discoverable; whole-row is faster. Revisit if user testing shows people accidentally dragging when trying to click. The existing single-click selection still works because zero-distance drag is a click, same pattern as Canvas.
- **What about display_name in the JSON for unset names?** Decision: **omit the key**. If displayName is undefined, sceneToJSON skips it. If it's an empty string, treat as undefined (sceneToJSON omits, jsonToScene that reads `""` would set undefined too). Empty-string display names are semantically pointless; collapsing them avoids round-trip drift.
- **File System Access feature flag.** What if the user is on a browser that has the API but it's disabled (e.g. enterprise policy)? `showSaveFilePicker()` will reject; surface as a banner error like a malformed file, then fall back. Decision: **don't auto-fall-back at runtime**. Instruct the user to use a browser that supports it; the fallback path stays for browsers without the API entirely (Safari, older Firefox). Why: silently falling back means the user thinks they're working with a real file when they're working with a download, which differs in the rewriting-the-same-file UX.
  - Counterpoint: the fallback IS what Safari users get permanently. Implementation reality: the `<a download>` path always creates a new file (with browser-decided destination). Document this in the README; don't try to paper over the gap.
- **Should drag-to-reorder live in EntityList or as a generic "Reorderable" component?** Decision: **inline in EntityList**. The pattern doesn't reuse anywhere else in the editor (Canvas doesn't reorder; assets compact automatically). Generalising for a single use is the wrong reflex.

## What ships at V2.4 closing

The cumulative acceptance test in `Requirements_V2.md` §Verification runs end-to-end. The editor builds, validates, saves, reloads, and re-imports a complete Engine.md §6 scene that the Python pipeline can compile and the C engine can run. Every Engine.md §13 rule is enforced, surfaced inline, and tested in `validateScene.test.ts`.

Open paths after V2 closes:
- **Desktop wrapper** (Tauri or otherwise). The `IOAdapter` and `AssetStore` interfaces are sized for a filesystem-backed swap; `runPipeline` lives behind a third interface that V2.4 does not implement. Land all three on the wrapper.
- **V3 polish** items enumerated in the per-slice "Explicitly NOT in V2.X" tables: auto-fix suggestions, recent files, asset bundling, custom keybindings, etc.
