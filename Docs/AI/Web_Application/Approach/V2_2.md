# V2.2 — Vertical Slice: Asset Pipeline (Sprites + Audio)

## Goal

Add the two engine components that reference assets (`sprite`, `audio`), and the asset store that backs them. After V2.2:

1. The user can drag-drop or click-pick a PNG into an "Assets" panel; it persists in IndexedDB and survives reload.
2. The same is true for 16-bit PCM WAVs.
3. Entities can have a `sprite` component; the canvas renders the texture inside the entity's transform, with `image-rendering: pixelated` and a `colour_tint` applied as an SVG filter (or fallback rect when the texture is missing).
4. Entities can have an `audio` component; the AssetBrowser exposes a "Preview" button per WAV that plays it through Web Audio.
5. Deleting a texture compacts every higher ID down by one and rewrites every sprite's `global_texture_id` accordingly, in a single undoable mutation. Same for audio.
6. The schema/Engine.md drift test grows two rows.

Validation UI (Engine.md §13's cross-component rules, density warnings, and dependency errors) lands in V2.3, not here.

---

## Why this slice

V2.2 is the only V2 slice that introduces a **new architectural surface**: asynchronous, persistent state outside `EditorCore`. Doing it as its own slice means:

- Async error modes (storage quota, parse failure, blob-URL leakage) land in one diff with the new code that triggers them.
- The AssetStore interface is designed once, not bolted on. The desktop wrapper later swaps the IndexedDB implementation for a filesystem-backed one without touching `core/` or `ui/`.
- The texture render path on the canvas has only one new abstraction at a time (the asset blob URL), not three (asset URLs, validation errors, JSON import paths).

Sprites and audio could in theory ship in two separate sub-slices (V2.2a, V2.2b) since they don't share fields. They share **all** the architecture they introduce — AssetStore, blob URLs, file dialogs, asset compaction — so splitting them creates two slices each carrying half the new concept. They land together.

## Build order

The order is chosen so each step keeps `npm run dev` runnable; UI integration comes last so async failure modes are exercised in tests before they hit the canvas.

1. **Define `AssetStore` interface in `core/AssetStore.ts`.** Pure types — no `idb` import. Methods:

   ```ts
   export interface AssetStore {
     getSnapshot(): AssetSnapshot
     subscribe(listener: () => void): () => void
     addTexture(record: NewAssetInput): Promise<AssetRecord>
     addAudio(record: NewAssetInput): Promise<AssetRecord>
     remove(kind: AssetKind, id: AssetId): Promise<void>
     getBlobURL(kind: AssetKind, id: AssetId): string | null  // null if not loaded yet or asset gone
   }
   export interface NewAssetInput { name: string; bytes: ArrayBuffer; originalFilename: string }
   export interface AssetRecord { id: AssetId; kind: AssetKind; name: string; bytes: ArrayBuffer; originalFilename: string; createdAt: number }
   export interface AssetSnapshot { textures: readonly AssetRecord[]; audios: readonly AssetRecord[] }
   ```

   Note: `subscribe`/`getSnapshot` mirror `EditorCore`'s contract so React's `useSyncExternalStore` works without adapter code. Snapshots are cached the same way (returned reference stable until next mutation).

   `getBlobURL` is synchronous because callers (Canvas render) need a URL on every paint. Implementation lazily creates blob URLs on first access and caches them, releasing on `remove`. **Never** generates a new URL per render — that would leak.

2. **Extend `FieldKind` with `asset-ref` and `colour-abgr`.** No schemas use them yet; tests stay green.

3. **Implement `IndexedDBAssetStore`.** Two object stores in one IDB database:

   - `textures` — keyPath `id`, autoincrement.
   - `audios` — keyPath `id`, autoincrement.

   The `id` exposed to consumers is the **dense rank** within the kind, not the IDB autoincrement. Store the IDB internal key alongside, but the public API uses the dense rank that stays stable until `remove`. On `remove(kind, k)`:
   - Delete row.
   - For every row with rank > k, decrement its rank.
   - Revoke the deleted blob URL.
   - Notify subscribers.

   **Important implementation detail:** the dense-rank model means `id` *changes* on remove. Consumers (sprite components) must update too — that's `EditorCore.removeAsset`'s job (step 5). The store provides the snapshot; the EditorCore propagates to entity refs.

   Tests against `fake-indexeddb`:
   - Add three textures, snapshot has IDs 0/1/2 in order added.
   - Remove ID 1, snapshot has IDs 0/1 (former 2 is now 1).
   - `getBlobURL` returns same string across calls until `remove` then is invalidated.
   - Reload (simulate by re-instantiating the store on the same DB) → snapshot reconstructs identical IDs. Persistence is the whole point.

4. **PNG and WAV validation utilities.**
   - `pngLimits.ts`: parse the PNG signature + IHDR chunk to extract width/height. Compute padded (next power of 2 in each dimension). Reject if either padded dim > 512. Pure function over ArrayBuffer; throws with a structured message naming the source dims and predicted padded dims.
   - `wavValidation.ts`: parse RIFF/WAVE header, find `fmt ` chunk, assert `wFormatTag === 1` (PCM) and `wBitsPerSample === 16`. Stereo/mono both pass — Engine.md §7 says converter downmixes; editor only rejects bits and format.

   Both have unit tests with hand-crafted minimal headers (or a tiny known-good fixture). Reject malformed bytes with clear messages — these surface in the AssetBrowser UI.

5. **`EditorCore.addTextureAsset` / `addAudioAsset` / `removeAsset`.** All async. None use the V2.0 `beginTransaction`/`commitTransaction` API — that API is sync-only and forbids nesting, neither of which fits async asset I/O. Instead, each method snapshots `{scene, assets}` *before* its async work, performs the mutation, then pushes one combined history entry on success. They throw immediately if `txOpen` is `true` (a sync transaction is in progress); the AssetBrowser disables its buttons via `useScene` selection on `txOpen` exposed through the snapshot to make the throw path unreachable in normal use.

   `addTextureAsset(file: File)`:
   - Read bytes (`file.arrayBuffer()`).
   - Validate via `pngLimits`. Throw on rejection (caller catches and surfaces).
   - Snapshot `prevScene = this.scene`, `prevAssets = this.assetStore.getSnapshot()`.
   - `await this.assetStore.addTexture({ name: deriveDisplayName(file), bytes, originalFilename: file.name })`.
   - Push history entry `{ scene: prevScene, assets: prevAssets }`. (Scene is unchanged by addTexture, but the entry's `assets` field is what undo restores.)
   - Emit. Return the new `AssetRecord` (with its assigned dense ID).

   Decision (see Open questions): asset additions DO push to history so that undo can remove the asset; otherwise the editor has a partial-undo trap where you can undo edits but not asset uploads.

   `removeAsset(kind, id)` is the heavyweight one:
   - Snapshot `prevScene`, `prevAssets`.
   - Compute the new scene with refs rewritten in one immutable pass: for every entity, for every component with an `asset-ref` field of the matching kind, set ref to `null` if `===id`, decrement if `>id`. One `entities` array rebuild, not per-entity history entries.
   - `await this.assetStore.remove(kind, id)`.
   - Set `this.scene = newScene`. Push history entry `{ scene: prevScene, assets: prevAssets }`. Emit.

   Tests:
   - Three sprites referencing tex 0/1/2; remove tex 1 → sprite refs become 0/null/1.
   - Single-step undo restores asset, refs, and selection.
   - `addTextureAsset` rejects when `pngLimits` throws; no history entry written.
   - Calling `addTextureAsset` while a sync transaction is open throws and writes nothing.

   Mechanics this requires from V2.0's history system: history entries become `{ scene: Scene; assets: AssetSnapshot }` instead of bare `Scene`. The asset store exposes a `restoreSnapshot(snapshot): Promise<void>` method that re-creates whatever the snapshot describes — re-adding deleted records, removing added ones, by id. `undo()` and `redo()` both `await` the asset restoration before emitting. This is documented explicitly in the V2.2 history test: "asset mutations are part of the same undo step as the scene changes they cause."

6. **Schemas: `sprite.ts` and `audio.ts`.**
   - `sprite`: `global_texture_id` (asset-ref texture, default `null` ≡ unset), `colour_tint` (colour-abgr uint32, default `0xFFFFFFFF`). `requires: ['transform']`. Serialize: `{ global_texture_id: id ?? -1, colour_tint }`. -1 represents unset on the wire; engine renders the coloured fallback per §5.2.
   - `audio`: `sound_id` (asset-ref audio, default `null`), `volume` (float 0–1, default 1.0), `loop` (bool, default false). No `requires`. Serialize: `{ sound_id: id ?? -1, volume, loop: loop ? 1 : 0 }`.

   The `null → -1` serialization is the single place this convention lives. Tests assert it both ways (null ↔ -1).

7. **Update `registry.drift.test.ts`** to include sprite (mask 2, size 8) and audio (mask 6, size 12) rows. Field names per Engine.md §5.2 / §5.6: `global_texture_id`, `colour_tint`; `sound_id`, `volume`, `loop`.

8. **PropertyPanel: asset-ref and colour-abgr branches.**
   - **asset-ref**: subscribe to AssetStore via `useAssets()` (see step 9). Render `<select>` with one `<option>` per asset of the right kind, plus a leading `<option value="">Unset</option>`. Value is the asset ID as a string; convert on change. Transactions still apply via the focus/blur wrapper from V2.0.
   - **colour-abgr** (extracted to `ColourTintField.tsx`): displays a `<input type="color">` for RGB and a separate `<input type="range" min={0} max={255}>` for alpha. The schema field stores a uint32; the component splits it on render and reassembles on change. ABGR byte order matches PSP convention: `(A << 24) | (B << 16) | (G << 8) | R`. Test the byte order with a fixture: `0xFFFFFFFF` → white opaque; `0xFF0000FF` → red opaque (R=0xFF, G=0x00, B=0x00, A=0xFF); etc.

9. **`useAssets()` hook.** Mirror of `useScene()` — three lines, `useSyncExternalStore(assetStore.subscribe, assetStore.getSnapshot)`.

10. **`<AssetBrowser>`.** Two tabs (Textures, Audio). Each tab:
    - "Add Texture" / "Add Audio" button opens a hidden `<input type="file" accept="image/png">` / `accept=".wav"`. On file pick, calls `core.addTextureAsset(file)` / `core.addAudioAsset(file)`. Errors surface as a dismissable banner above the list.
    - List of assets: thumbnail (texture: `<TextureThumbnail>`; audio: filename + bytesize) + Delete button. Texture uses `assetStore.getBlobURL('texture', id)` as `<img src>`.
    - Audio: a "Preview" button that calls into `audioContext.ts` to play. Implementation: lazy `AudioContext`, decoded buffers cached by id, one-shot `BufferSource` per click. `audioContext.ts` exposes `playAudio(id, bytes)` and resolves the cached/decoded buffer.

11. **Canvas texture render.** Inside the `<rect>` map, when an entity has `sprite` and the referenced texture exists, render an `<image>` with `href={blobUrl}` instead of (or above) the coloured rect. CSS sets `image-rendering: pixelated` (already on `.canvas`). When the ref is unset or the asset has been deleted, fall back to a `<rect>` filled with `#colourTintCSS(sprite.colour_tint)` — same byte-decode as the picker but emitting a CSS rgba string.

    The selected-state outline still applies to the rect (or the image's stroke via SVG); keep `data-selected` on the outermost element of the entity render.

12. **App layout grows a fourth region.** Sidebar (entity list), canvas, panel, asset browser. The asset browser fits as either a fourth column or a panel below the sidebar. **Decision for V2.2: below the sidebar**, sharing the left column. The canvas wants horizontal real estate; asset thumbnails are square-shaped and don't.

    `index.css` updates the `.layout` flexbox accordingly; `.sidebar` becomes a flex column with `<EntityList>` on top and `<AssetBrowser>` below.

13. **`main.tsx` composition.** Build the `IndexedDBAssetStore` (returns a `Promise<AssetStore>` since IDB open is async), then `new EditorCore(assetStore)`, then `<App core={core} assetStore={assetStore} />`. Show a brief "Loading…" placeholder while the IDB connection resolves. If the connection fails (private mode), render the editor with an in-memory fallback `AssetStore` and a banner explaining persistence is off.

Steps 1–7 are pure logic and unit-tested. Steps 8–12 are UI and rely on manual checks. Step 13 wires composition and gates on the loading state.

## Open questions to resolve during V2.2

- **Where do asset additions sit in the undo history?** Decision: **inside it**. Adding a texture and immediately undoing should revert to the prior asset state. The alternative (asset adds are non-undoable) creates an inconsistent UX where some user-visible state survives Cmd+Z and some doesn't. Implementation: `EditorCore` snapshots `assetStore.getSnapshot()` alongside `Scene` per history entry; undo restores both via `assetStore.restoreSnapshot()`.
- **`null` vs `-1` for unset asset refs.** Internal representation: `null`. JSON wire: `-1` (an int the engine treats as "no asset"). Documented in `sprite.ts` and `audio.ts` headers; round-trip tests assert both directions.
- **Should `removeAsset` need user confirmation?** V2.2: **no inline confirmation** — undo is one keystroke away. If field-testing shows accidental deletes are common, V2.3 can add an inline "n entities reference this" warning and an Undo toast. Don't pre-emptively build a confirmation dialog.
- **Asset names.** Editable inline in V2.2? Decision: **no**. The display name is derived from the original filename (stripped of extension). Renaming is V3 polish. The "name" field on `AssetRecord` exists so V3 has somewhere to store it.
- **AudioContext lifecycle.** Decision: a single lazily-constructed `AudioContext` shared across the editor. Browsers require a user gesture to start it; `audioContext.ts` ensures the first call is from a click handler (Preview button) and resumes if suspended. No autoplay, no preload — the engine is the play-target, not the editor.
- **What if IndexedDB is blocked (private mode)?** Decision: fallback to an in-memory `AssetStore` with the same interface; show a persistent banner. The user can still build a scene and export `scene.json`; assets just won't survive reload. The non-blocking degradation matters because Safari aggressively blocks IDB in some contexts.
- **Texture render: `<image>` inside the same `<svg>`?** Yes. SVG `<image>` honours `image-rendering: pixelated`. Alternative (HTML `<img>` overlaid in absolute position) means dual coordinate systems and more drift between drag math and render. The drift test wins again — keep one coordinate space.
- **Asset compaction vs export-time compaction.** Decided in `Requirements_V2.md`: editor enforces dense-from-0 by auto-compaction. `removeAsset` rewrites refs in the same mutation; one history entry per delete. JSON output is always dense.
- **When the user uploads a duplicate file (same bytes).** Decision: **don't dedupe**. Two distinct assets with identical bytes is fine; Engine.md doesn't constrain content uniqueness. Dedupe by hash is V3 polish; pre-emptive optimization for a problem nobody has yet.
- **Audio entities without a `transform`.** Engine.md doesn't require it (§5.6 doesn't list `requires`). V2.2's `audio` schema doesn't declare `requires`. The `Empty` preset + `addComponent('audio')` is the official path; the AssetBrowser doesn't need its own "Add Audio Entity" shortcut.

## What V2.3 unlocks (the next slice after V2.2)

- Validation now has real cross-component rules to enforce: sprite refs that are unset, texture-id density (already enforced by compaction, but the rule belongs in the validator for completeness), `player_controlled ⇒ physics + transform`, `sprite ⇒ transform`, `collider ⇒ transform`, `physics ⇒ transform`, entity count ≤ 256, total texture footprint ≤ 8MB warning.
- `validateScene(scene, assets) → ValidationIssue[]` is a pure function over the snapshot pair; no IO, no async. It plugs into `useScene` + `useAssets` and renders inline on the panel and as a toolbar status badge.
- The Canvas's existing coloured-fallback rect for missing textures (added in V2.2 to mirror engine behaviour) now also gets a per-entity inline error via the panel — same data path, two presentations.
