# V2.3 — Vertical Slice: Validation UI

## Goal

Make every Engine.md §13 rule visible to the user the instant it fires. After V2.3:

1. A toolbar badge reads either "Valid," "n warnings," or "n errors." It updates synchronously on every mutation.
2. The entity list shows a per-entity error/warning marker; selecting an entity with errors highlights the offending field in the property panel inline.
3. The "Show JSON" / "Save" buttons (V2.4 introduces Save proper; V2.3 still shows the V1.5 dialog) are disabled when there are any **errors**. Warnings don't block export.
4. Specific Engine.md §13 rules are surfaced with actionable messages: "sprite references missing texture," "player_controlled requires physics," "PNG would pad to 1024×512 (max 512×512)" (texture upload time, already wired in V2.2 — V2.3 just verifies it integrates).

No new components. No new IO. No data-shape changes — V2.3 adds a new pure function and renders its output.

---

## Why this slice

Validation has nothing to validate without the full schema set + assets. Engine.md §13 rules cite:
- entity count, sprite refs, audio refs, ID density, padded size, WAV format → need V2.2's AssetStore.
- `player_controlled ⇒ physics`, `sprite ⇒ transform`, etc. → need V2.1's component schemas.

Doing validation as its own slice means the rule-by-rule coverage is line-itemized in one diff (every rule in Engine.md §13 maps to a named test). It also keeps the V2.2 diff focused on async / persistence and prevents validation logic from creeping into the AssetStore as side-effects.

There's a temptation to merge V2.3 into V2.4. Don't: V2.4's import path actively *needs* validation to gate "is this scene safe to load," and bundling them makes the import diff carry validator changes too.

## Scope

### In V2.3

| Layer       | What                                                                                                                                                  |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| Core        | `core/validation/validateScene.ts` — pure `(scene: Scene, assets: AssetSnapshot) => ValidationIssue[]`. No imports from `io/` or `ui/`. Imports from `schemas/` (registry-aware rules) and `core/types`. |
| Core types  | `ValidationIssue` already declared in V1.1 types.ts — re-used. Adds optional `code: string` (machine-readable, e.g. `'sprite-requires-transform'`) so tests assert on codes, not message text. |
| UI          | `useValidation()` hook: derives `ValidationIssue[]` from `useScene()` + `useAssets()`, memoized by snapshot reference equality (cached snapshots from V2.0/V2.2 make this cheap). |
| UI          | `<ToolbarStatus>` — badge in the existing toolbar, colour-coded green/yellow/red. Click expands a popover listing all issues, grouped by severity, each clickable to focus the offending entity / field. |
| UI          | `<EntityList>` rows with issues get a small icon (red dot for errors, yellow for warnings). Hover shows the count. ARIA: `aria-describedby` linking to a hidden text node listing issue messages, so screen readers announce them. |
| UI          | `<PropertyPanel>` field rows with issues get a red border (CSS via `data-error`) and an inline message below the input. Section headers with section-level issues (e.g. "this component has missing prerequisites") show the same. |
| UI          | "Show JSON" button disabled when any error issue exists; tooltip explains why. |
| Tests       | One spec per Engine.md §13 rule. Each builds a minimal scene that violates the rule, asserts `validateScene` returns the expected code(s). The validator's test file is the line-item Engine.md §13 coverage map. |

### Explicitly NOT in V2.3 (deferred)

| Deferred to | What |
| ----------- | ---- |
| **V2.4**    | Disabling Save/Export-with-Save and gating Import on the validation result. V2.3 disables "Show JSON" only because that's the only export gesture that exists today. V2.4 generalizes when Save/Load buttons land. |
| **V3**      | Auto-fix suggestions ("Add `transform` to fix this"). V2.3 surfaces the problem; the fix is one click in the existing UI. |
| **V3**      | Validation rule customization or per-project disabling. Engine.md §13 rules are the engine's contract — they're not optional. |
| **V3**      | Performance budget for validation on huge scenes (256-entity edge of MAX_ENTITIES). Synchronous validation on every mutation is fine at this scale; profiling is a V3 concern if it becomes one. |

## Engine.md §13 rule → validator code mapping

This table is the spec for `validateScene.test.ts`. One test per row.

| Rule (Engine.md §13)                                                              | Code                                | Severity | Source data        |
| --------------------------------------------------------------------------------- | ----------------------------------- | -------- | ------------------ |
| Entity count ≤ 256                                                                | `entity-count-exceeded`             | error    | `scene.entities.length` |
| Every sprite's `global_texture_id` points to a loaded asset                       | `sprite-texture-missing`            | error    | sprite refs vs `assets.textures` |
| Every audio's `sound_id` points to a loaded WAV                                   | `audio-sound-missing`               | error    | audio refs vs `assets.audios` |
| Texture IDs dense from 0 (no gaps)                                                | `texture-ids-not-dense`             | error    | `assets.textures` IDs |
| Sound IDs dense from 0                                                            | `audio-ids-not-dense`               | error    | `assets.audios` IDs |
| No PNG pads past 512×512                                                          | (handled at upload time in V2.2; this rule does not appear in `validateScene`. The text is in Engine.md §13 but the editor enforces it earlier; documented in `validateScene.ts` header.) | n/a | n/a |
| WAVs are 16-bit PCM                                                               | (handled at upload, same as above)  | n/a      | n/a |
| `player_controlled ⇒ physics`                                                     | `player-controlled-requires-physics`  | error    | scene |
| `player_controlled ⇒ transform`                                                   | `player-controlled-requires-transform`| error    | scene |
| `sprite ⇒ transform`                                                              | `sprite-requires-transform`         | error    | scene |
| `collider ⇒ transform`                                                            | `collider-requires-transform`       | error    | scene |
| `physics ⇒ transform`                                                             | `physics-requires-transform`        | error    | scene |
| Total texture footprint ≤ 8 MB                                                    | `texture-footprint-warning`         | warning  | sum of padded RGBA8888 sizes per texture |

The two upload-time rules (PNG padded size, WAV bit depth) are deliberately NOT in `validateScene`. They run at upload time in V2.2 because by the time an asset is in the store, it's already validated; re-checking on every mutation is wasted work. The validator file's header comment links to the upload-time rules so the Engine.md §13 reader knows where every rule lives.

`requires`-based rules duplicate what V2.1's `addComponent` enforces. The validator catches the case where a component was removed in a way that orphans a peer (V2.1 disables the remove button; the validator is belt-and-braces, and also catches imported scenes from V2.4 that violate the rule).

## Build order

Each step keeps everything green. UI work last because the validator and tests are pure logic.

1. **Add `code` to `ValidationIssue`.** Update `core/types.ts`. Existing `validate()` methods on schemas (which return `[]` today) are unaffected — `code` is optional.

2. **Implement `validateScene.ts`.** Layout:

   ```ts
   export function validateScene(scene: Scene, assets: AssetSnapshot): ValidationIssue[] {
     const issues: ValidationIssue[] = []
     issues.push(...checkEntityCount(scene))
     issues.push(...checkRequires(scene))
     issues.push(...checkSpriteRefs(scene, assets))
     issues.push(...checkAudioRefs(scene, assets))
     issues.push(...checkAssetIdDensity(assets))
     issues.push(...checkTextureFootprint(assets))
     return issues
   }
   ```

   Each `check*` is a small pure function. One test file describes per rule:

   ```ts
   describe('player_controlled requires physics', () => {
     it('flags an entity with player_controlled but no physics', () => { ... })
     it('does not flag when physics is present', () => { ... })
   })
   ```

   The test file is the implementation spec — every `it` corresponds to a row in the table above plus its negative case.

3. **`useValidation()` hook.**

   ```ts
   export function useValidation(core: EditorCore, assetStore: AssetStore): ValidationIssue[] {
     const scene = useScene(core)
     const assets = useAssets(assetStore)
     return useMemo(() => validateScene(scene, assets), [scene, assets])
   }
   ```

   Memoization keyed on snapshot identity (cached references from V2.0/V2.2 make this hold). The hook is dirt-simple; the architecture earned this in V2.0.

4. **`<ToolbarStatus>` badge.** Counts errors and warnings, renders a coloured pill:
   - `0/0` → green "Valid"
   - `0 errors / N warnings` → yellow "N warnings"
   - `N errors / M warnings` → red "N errors, M warnings"

   Click expands a `<details>` (or popover) listing every issue grouped by severity. Each issue is a button that, on click, calls `core.selectEntity(issue.entityId)` if set, and if `componentKey`/`fieldName` are set, briefly highlights that section in the panel via a transient "focused" state (CSS animation, not state in `EditorCore` — purely UI sugar).

5. **EntityList row indicator.** For each entity, count issues with `entityId === entity.id`. Render a dot beside the row label, coloured by max severity. Use `title` for the count; ARIA via a visually-hidden `<span>` so screen readers get "Entity #1, 2 errors" inline.

6. **PropertyPanel field-level rendering.** Pass `issues` (filtered to the selected entity) into `ComponentFields`. For each field, find issues where `componentKey` and `fieldName` match. If any, set `data-error` on the input wrapper and render an inline `<small>` below it.

   Section-level issues (e.g. `player-controlled-requires-physics` has `componentKey === 'player_controlled'` but `fieldName` is null) render at the section header.

7. **Disable "Show JSON" on errors.**

   ```tsx
   <button disabled={issues.some(i => i.severity === 'error')} ...>Show JSON</button>
   ```

   Tooltip via `title`: "Fix N errors before exporting." Warnings don't block. The disabled state itself is the user-facing invariant the validator's `code` set protects.

Steps 1–2 are the spec; step 3 wires it; 4–7 render. None require coordination with V2.4.

## Acceptance criteria

- `npm run typecheck && npm run lint && npm run test` — clean. Validator test file enumerates every Engine.md §13 rule that runs in the validator; the upload-time rules are handled in V2.2's tests.
- Manual checks (`npm run dev`):
  - Fresh editor with no entities → toolbar reads "Valid." Show JSON enabled.
  - Add a Dynamic entity, then remove its `transform` (V2.1 disables this if `physics`/`collider` depend, but you can remove `physics` first, then `collider`, then `transform`) → no transform-dependent components left, "Valid" again. Re-add `physics` first → toolbar flips to "1 error" (`physics-requires-transform`); panel highlights the Physics section header in red.
  - Add a sprite to an entity, leave its texture unset → "1 error" (`sprite-texture-missing`); panel shows red border on the Texture dropdown with inline message. Pick any texture → error clears synchronously.
  - Upload three textures, then delete the middle one (V2.2 already compacts refs, so the only path to a non-dense state is `assets` constructed by V2.4 import; for V2.3 manual smoke, this rule is exercised purely by tests).
  - Toggle `player_controlled` on an entity that has `transform` but no `physics` → "1 error" (`player-controlled-requires-physics`); Show JSON button greys out, hover tooltip says "Fix 1 error before exporting." Add `physics` → error clears, button re-enables.
  - Click the toolbar badge → popover lists every issue; click a listed issue → corresponding entity selects, panel scrolls/highlights the field.
  - Cmd+Z over a sequence of changes that toggle errors → toolbar badge updates step by step, each undo's badge state matches the historic state.

## Open questions to resolve during V2.3

- **Where does the texture-footprint computation live?** Decision: **inside the validator**, taking padded dimensions from each `AssetRecord` (V2.2 already validates the padded size on upload, so this is just a sum). The 8 MB threshold is the Engine.md §2 "warning" budget; over it surfaces as warning severity, not error — the user might intentionally push the limit.
- **Should `validateScene` be deduplicating issues?** No. If three sprites all reference missing texture 5, that's three issues (one per entity). Combining hides the per-entity locality.
- **Should the validator include `display_name` warnings?** Decision: **no**. Per Engine.md §6 the compiler ignores display_name; an entity with no display name is fine. V2.4 introduces the editing UI; a missing display_name is a UI nicety, not a contract violation.
- **Performance.** With 256 entities × ~7 rules each + asset checks, worst case is ~1800 issue computations per mutation. None of the rules are quadratic; total is microseconds. No optimization needed; the V3 budget reminder exists if it ever does.
- **Severity for ID-not-dense in normal use.** V2.2's `removeAsset` rewrites refs to keep IDs dense, so this rule should never fire in normal operation. It exists for V2.4's import path: a hand-edited `scene.json` with sparse texture IDs gets caught and surfaced. Keep severity at error so import-of-broken-scenes doesn't silently load.
- **What about issues without an entityId (e.g. entity-count-exceeded)?** They render in the toolbar popover but not in the EntityList. Section header in the popover is "Scene-level."

## What V2.4 unlocks (the next slice after V2.3)

- Save/Load buttons can gate on the same `issues.some(i => i.severity === 'error')` predicate without re-deriving anything.
- Imported `scene.json` runs through `validateScene` immediately after `jsonToScene`; failures don't block import (they'd block a hand-typed file from ever loading) but the imported state lights up with errors so the user sees what's wrong.
- The full V2 acceptance test in `Requirements_V2.md` becomes runnable: build the Engine.md §6 scene, observe "Valid" in the toolbar, save, reload, open, observe "Valid" again. End-to-end.
