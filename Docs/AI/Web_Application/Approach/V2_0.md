# V2.0 — Vertical Slice: Undo / Redo Foundation

## Goal

Add undo and redo to the editor before any new mutation methods land in V2.1+. After V2.0:

1. Cmd+Z (Ctrl+Z on Linux/Windows) reverts the most recent edit and re-emits to subscribers.
2. Cmd+Shift+Z (Ctrl+Shift+Z) replays the most recently undone edit.
3. Dragging a rect on the canvas is **one** undo step, not 100s of pointer-move steps.
4. Editing a number input is **one** undo step per focus session, not one per keystroke.
5. Adding an entity, then editing its X, then undoing twice → entity is gone, panel shows "no selection," entity list empty.

No new components, no new field kinds, no new IO. The acceptance criteria are entirely about history mechanics on the V1.5 mutation surface.

---

## Why this slice (and why first)

V1's Requirements.md flagged undo/redo as "the single highest-leverage decision in the plan to reconsider before code starts." The retrofit cost was estimated at multi-day if commands had to be threaded through every existing mutation. With the immutable `Scene` snapshots that `EditorCore` already produces, the implementation is much smaller — store the prior `Scene` reference on every mutation; undo is "pop and assign."

Doing it **before** V2.1+ means every new mutation method (`addComponent`, `removeComponent`, `addAsset`, `removeAsset`, `moveEntity`, `setDisplayName`) joins the history system by being written through it from day one. Doing it after means six retrofits in a row. The math doesn't favour deferral.

The choice between **snapshot history** and **command pattern** is decided in the V2 requirements doc; V2.0 just implements the snapshot side.

## Scope

### In V2.0

| Layer       | What                                                                                                                                            |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| Core        | `EditorCore` gains `undo()`, `redo()`, `beginTransaction()`, `commitTransaction()`, `abortTransaction()`, `canUndo`/`canRedo` getters on the snapshot. |
| Core types  | `Scene` shape unchanged. `EditorSnapshot` (renamed from the bare `Scene` returned by `getSnapshot`) gains `canUndo: boolean` and `canRedo: boolean` for `useScene`-driven UI to read. |
| UI          | Toolbar gains Undo / Redo buttons, disabled when respective stack empty. Canvas drag wraps `onPointerDown` → `beginTransaction`, `onPointerUp`/`onPointerCancel` → `commitTransaction`. PropertyPanel `<input>` wraps focus → `beginTransaction`, blur → `commitTransaction`. |
| UI          | Global keyboard handler (mounted in `App.tsx`): Cmd/Ctrl+Z → `core.undo()`, Cmd/Ctrl+Shift+Z → `core.redo()`. Suppressed when focus is inside an `<input>` or `<textarea>` so native input undo still works (browsers maintain their own per-input undo stack). |
| Tests       | New `EditorCore.history.test.ts`: undo restores prior scene, redo replays, transaction collapses N internal mutations to one history entry, abort discards the transaction, undo after a fresh mutation clears the redo stack, history depth cap behaves. |

### Explicitly NOT in V2.0 (deferred)

| Deferred to | What |
| ----------- | ---- |
| **V2.1+**   | New component schemas / new field kinds. Their mutation methods inherit the transaction system but they're written in their own slices. |
| **V2.2+**   | `AssetStore` mutations participating in undo. `addAsset` / `removeAsset` open transactions when V2.2 lands; the history mechanism does not change. |
| **V3**      | Per-input keystroke coalescing beyond the focus session (e.g. typing "100" then immediately typing "200" without losing focus is one history entry, not two — which is correct, by design, since they are one focus session). |
| **V3**      | History depth tuning, persistent history across reloads, history inspector. V2.0 ships with a fixed cap (see Open questions) and an in-memory stack only. |
| **V3**      | Custom keybindings. Cmd+Z / Cmd+Shift+Z / Ctrl variants are hardcoded. |
| **V3**      | Multi-document undo. Single-scene editor only. |

## Build order

Each step is independently runnable. After every step `npm run typecheck && npm run lint && npm run test` exits clean and `npm run dev` still demos V1.5 behaviour.

1. **Extract `EditorSnapshot` type and rename the return of `getSnapshot()`.** `core/types.ts` declares:

   ```ts
   export interface EditorSnapshot extends Scene {
     canUndo: boolean
     canRedo: boolean
   }
   ```

   `EditorCore.getSnapshot(): EditorSnapshot`. V1.5 tests that read scene fields keep working (structural extension); UI consumers get the two new fields for free. Update `useScene.ts`'s return type.

   At this step `canUndo` and `canRedo` are always `false`. Tests stay green.

2. **Add the history stack and `undo`/`redo`, no transactions yet.** Inside `EditorCore`:

   ```
   private history: Scene[] = []
   private redoStack: Scene[] = []
   private readonly historyCap = 100
   ```

   Add a private `pushHistory(prev: Scene)` called at the top of every mutation that already exists (`addEntity`, `selectEntity`, `setField`, `setFields`). It:
   - pushes `prev` onto `history`,
   - clears `redoStack`,
   - if `history.length > historyCap`, drops the oldest entry.

   `selectEntity` is a special case: it mutates `Scene` (selection is part of the snapshot) but should arguably not be undoable. Decision: **selection IS undoable** — it's part of the `Scene` and the simplest rule wins; the alternative is an exception for one method that complicates the contract. If user feedback later asks for "undo skips selection," revisit; cheap to change.

   `undo()`:
   - if `history.length === 0` return,
   - push `this.scene` onto `redoStack`,
   - `this.scene = history.pop()!`,
   - emit.

   `redo()`: symmetric.

   `getSnapshot()` returns `{ ...scene, canUndo: history.length > 0, canRedo: redoStack.length > 0 }`.

   **Identity caveat** — `getSnapshot()` previously returned the same `Scene` reference between mutations, which `useSyncExternalStore` relies on to skip rerenders. Wrapping in a fresh object every call breaks that contract. Fix: cache the snapshot, invalidate on every emit.

   ```
   private cachedSnapshot: EditorSnapshot | null = null
   getSnapshot = (): EditorSnapshot => {
     if (this.cachedSnapshot === null) {
       this.cachedSnapshot = { ...this.scene, canUndo: …, canRedo: … }
     }
     return this.cachedSnapshot
   }
   private emit = () => {
     this.cachedSnapshot = null
     for (const l of this.listeners) l()
   }
   ```

   This keeps reference equality under React's `useSyncExternalStore` `getSnapshot` contract (must return the same value if state hasn't changed).

   At this step the V1.5 tests still pass. New test: add entity → undo → entity gone; redo → entity back. Mutating after undo clears redo stack.

3. **Add the transaction API.** Three methods on `EditorCore`:

   ```ts
   beginTransaction(): void   // throws if already in a transaction
   commitTransaction(): void  // throws if no open transaction
   abortTransaction(): void   // throws if no open transaction
   ```

   Internal state:

   ```
   private txOpen = false
   private txStartScene: Scene | null = null
   ```

   `beginTransaction`: snapshot `this.scene` into `txStartScene`, set `txOpen = true`. Does **not** touch history yet.

   `pushHistory(prev)` becomes a no-op while `txOpen` — mutations during a transaction don't grow history one-per-call. They still mutate `this.scene` and emit (so the canvas updates live during a drag).

   `commitTransaction`: if `txStartScene !== this.scene` (i.e. something actually changed), push `txStartScene` onto history exactly once, clear redo stack. Otherwise do nothing (an empty transaction — e.g. user pressed pointerdown and released without moving). Reset `txOpen = false`, `txStartScene = null`. Emit so `canUndo` flips for the toolbar.

   `abortTransaction`: `this.scene = txStartScene`, reset, emit. Used on pointercancel — mid-drag the user yanks focus or the OS cancels the touch.

   No nesting in V2.0. Calling `beginTransaction` while `txOpen` throws — V2.1+ slices that need composed mutations will introduce a guarded helper (`runInTransaction(fn)`); the explicit pair stays for UI-driven gestures.

   New tests:
   - `setField` × 3 inside a transaction → undo restores the pre-`begin` value (one entry).
   - empty transaction (begin → commit with no mutations) → history unchanged.
   - aborted transaction → scene reverts to pre-begin; history unchanged; redo stack unchanged.
   - mutations outside any transaction still produce one history entry per call (V1.5 behaviour preserved).

4. **Wire Canvas drag to transactions.** In `Canvas.tsx`:
   - `onPointerDown` calls `core.beginTransaction()` immediately after `core.selectEntity(entityId)`. The select-before-drag sequence becomes one transaction containing both the selection and the position deltas; undo collapses the entire interaction to one step.
   - `onPointerUp` and `onPointerCancel` call `core.commitTransaction()` (or `abortTransaction()` on cancel — see Open questions).
   - If pointer capture is somehow lost without an up/cancel event, the editor is in a stuck-transaction state. Mitigation: pointerup handler is on the same element that captured the pointer; Web standard guarantees release. Don't paper over impossibilities with abort-on-blur; if it bites, fix the underlying.

   Manual verify: drag rect 100px, release, press Cmd+Z → rect snaps back to its pre-drag position in one step.

5. **Wire PropertyPanel inputs to transactions.** In `PropertyPanel.tsx`:
   - `<input>`'s `onFocus` (which already exists for the focused-ref guard) also calls `core.beginTransaction()`.
   - `onBlur` (which already commits the parsed value) also calls `core.commitTransaction()` after the `setField` call.
   - Each keystroke that calls `setField` during the focused session goes into the open transaction — no per-keystroke history entries.
   - If the user's blur path hits the "invalid input → snap back to old value" branch, the transaction is committed empty (since `setField` was called with the old value, which short-circuits in `setFields` only if the partial differs… actually `setFields` always replaces; it doesn't dedupe). Decision: explicit `commitTransaction()` either way; an empty-result transaction is a no-op per step 3. Don't add deduplication to `setFields`.

   Manual verify: focus X input on selected entity (defaults to `0`), type `100`, blur, Cmd+Z → X is `0` again, panel reflects.

6. **Add Undo/Redo buttons + global keyboard shortcut.**
   - Toolbar in `App.tsx` gains two buttons: `disabled={!snapshot.canUndo}` / `disabled={!snapshot.canRedo}`. Click → `core.undo()` / `core.redo()`. Reuse existing `.toolbar button` styles.
   - New hook `ui/keybindings/useUndoRedoShortcuts.ts` mounts a single `keydown` listener on `window`. Logic:

     ```ts
     if (e.key === 'z' && (e.metaKey || e.ctrlKey)) {
       const target = e.target as HTMLElement | null
       if (target && (target.tagName === 'INPUT' || target.tagName === 'TEXTAREA' || target.isContentEditable)) return
       e.preventDefault()
       if (e.shiftKey) core.redo()
       else core.undo()
     }
     ```

     Bailing out for `<input>`/`<textarea>`/contenteditable preserves browser-native undo inside text fields. The PropertyPanel transaction system makes the *editor* undo on Cmd+Z work too, but only after the input loses focus — that's the expected mental model (in-progress text is a text-edit, committed fields are an editor-edit).

   - `App.tsx` calls the hook once. Hook owns add/remove via `useEffect`.

   Manual verify: click in canvas, Cmd+Z undoes; click into number input, Cmd+Z behaves like a normal text-input undo (last keystroke), blur and Cmd+Z again undoes the whole edit.

7. **Cap history at 100 entries.** Already implemented in step 2; explicit test asserts oldest entry drops when `history.length` would exceed `historyCap`. 100 is generous for an interactive session and bounded enough to keep memory predictable (256-entity scene ≈ ~50KB JSON; 100 entries ≈ 5MB worst case, fine).

Steps 1–3 are pure logic and unit-tested. Steps 4–6 are UI and rely on manual checks below plus the existing Canvas/Panel tests staying green.

## Acceptance criteria

- `npm run typecheck` — clean.
- `npm run lint` — clean. Boundary rules unchanged.
- `npm run test` — all V1.5 tests still green; new `EditorCore.history.test.ts` green (transactions, abort, redo-clear-on-mutation, history cap).
- `npm run dev` — manual checks:
  - Add entity → "Add Entity" appears in toolbar history (Undo enabled). Click Undo → entity gone, list empty, panel shows "no selection."
  - Add two entities, drag one 50px right, edit Y to `100` in the panel. Press Cmd+Z three times → Y back to `0`, drag undone, second entity gone (in that order). Press Cmd+Shift+Z three times → all three replayed in original order.
  - Drag a rect, mid-drag press Esc (which would normally fire `pointercancel`)… actually on most browsers Esc doesn't cancel pointer; this is just a manual check that the *commit* path works. The cancel path is exercised by the test, not the manual check.
  - Type into the X input: keystrokes `1`, `2`, `3`. Single Cmd+Z **after blur** → X back to its pre-edit value in one step.
  - With focus inside the X input, hit Cmd+Z → browser undoes the last *character* (native input behaviour); editor history is unchanged.
  - Make some edits, undo a few, then make a new edit → Redo button greys out (redo stack cleared). Cmd+Shift+Z is a no-op.
  - Spam Cmd+Z past the start of history → no crash, Undo button stays disabled at empty history.

## The one new pattern this slice introduces

**Transactional mutations via begin/commit on a non-React store.** Most React state libraries expose batching as either implicit (microtask-coalesced) or explicit (`store.batch(fn)`). The choice in V2.0 is **explicit pair**, not a function-callback `runInTransaction(fn)`, because the lifetime is driven by user input events (pointerdown → pointerup, focus → blur) that can't be wrapped in a single sync function. The API matches the input lifecycle directly.

The non-obvious part: `beginTransaction` snapshots the *entire* `Scene`, not a "diff." On commit, we push that snapshot to history; mid-transaction the live `Scene` already advanced through N replacements. Undo restores the pre-begin reference. Memory is bounded by the snapshot cap, not by the depth of the transaction.

## Open questions to resolve during V2.0

- **Should `selectEntity` be undoable?** Decision: **yes** — it's a `Scene` field, the same mutation pipeline, and the simplest rule. Spec: "selecting then deselecting is two history entries; undo brings the prior selection back." If users complain that Cmd+Z undoing a click-to-select feels weird, revisit by skipping `selectEntity` from `pushHistory`. Don't optimize for this complaint pre-emptively.
- **Pointer cancel: commit or abort?** Decision: **abort**. `pointercancel` fires when the OS yanks the pointer (touch interrupted, browser decides to scroll, etc.). The user did not finish the gesture; restoring pre-drag state matches every other editor's behaviour. The drag handler in V1.5 currently treats cancel and up identically; V2.0 splits them.
- **Empty transactions (begin → commit with no mutations).** Decision: **no-op on history**, but **emit anyway** so the toolbar's `canUndo` recomputes if it changed. Keeps the contract uniform: every commit emits exactly once.
- **Nested transactions.** Decision: **disallow in V2.0** (throw). Composed mutations in V2.1+ wrap themselves in a single transaction, never call begin twice. If a future use case genuinely needs nesting (it shouldn't — this is a single-actor editor), introduce a counter, not a stack.
- **Selection persistence across undo.** When undo restores a `Scene` whose `selectedEntityId` references an entity that no longer exists *in that snapshot*, the panel correctly shows "no selection" because PropertyPanel does `find(e => e.id === id) ?? undefined`. No special handling needed.
- **History cap value.** Decision: **100**, in-memory. Tunable later; not configurable for end users in V2.0. The risk of a 100-entry cap eating a multi-hour edit session is negligible compared to the risk of unbounded growth in long-running sessions on memory-constrained devices.
- **Should `undo()` clear the focused input's transaction?** Edge case: user is typing in X (transaction open), then triggers Cmd+Z via toolbar click (focus moves to button → blur fires on input → transaction commits → undo runs). The transaction is committed naturally before undo executes; no special handling. Manual smoke-tested in step 6.

## What V2.1 unlocks (the next slice after V2.0)

- All future mutation methods (`addComponent`, `removeComponent`, `addAsset`, `removeAsset`, `moveEntity`, `setDisplayName`) inherit the transaction system. They wrap their internal sub-mutations in a single transaction so undo collapses them to one step — no per-method history bookkeeping.
- The `EditorSnapshot` type is now the canonical store shape; V2.1's new schemas don't change it.
- Cmd+Z works for everything the editor lets the user do, by construction. V2.1+ specs don't have to re-prove undo on every new mutation — only that the new mutation is correctly transactional, which is one assertion per spec.
