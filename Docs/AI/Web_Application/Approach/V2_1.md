# V2.1 — Vertical Slice: Schema-Only Components + New Field Kinds + Drift Test

## Goal

Add the three engine components that don't need assets: `collider`, `physics`, and `player_controlled` (flag). Extend the property-panel field renderer with `bool` and `enum` kinds (everything `physics` and `collider` need). Introduce the **schema/Engine.md drift test** that the V1.5 deferred until ≥2 schemas existed — there will now be four.

By the end of V2.1:

1. The toolbar's "Add Entity" button is replaced by a small menu of presets ("Empty," "Static Solid," "Dynamic"); each preset is a defined set of component keys.
2. The Property Panel for a selected entity shows an "Add Component" dropdown listing the registered components the entity doesn't already have. Each section has a "Remove" button on its header (greyed out when removal would violate `requires`).
3. Components with `bool` or `enum` fields render correctly: checkbox for bool, `<select>` for enum.
4. `player_controlled` is added as an "Add Flag" entry — toggling it flips an empty-body component on the entity and emits as a top-level boolean in `scene.json`.
5. `npm run test` runs the drift test; flipping `transform.maskBit` to `99` in the schema fails the test before any other behaviour breaks.

No assets. No validation UI yet. Sprite and audio still don't exist.

---

## Why this slice

Three of the five remaining components are entirely about scalar fields (no asset references, no async I/O). They're the cheapest way to verify that:

- The schema-driven panel renders without modification when the registry grows.
- `sceneToJSON` already handles arbitrary registry contents (it does, per V1.5; this proves it on production data).
- Composed mutations (`addComponent`, `removeComponent`) work correctly through the V2.0 transaction system.

V2.2's asset components (`sprite`, `audio`) are heavier and bring in a new architectural surface (`AssetStore`); doing them after V2.1 means asset bugs aren't tangled with field-kind bugs, and the drift test already exists to catch any layout drift introduced by the asset schemas.

The drift test earns its keep here. With four schemas (V2.1's three plus V1.5's `transform`) the table is big enough that hand-written assertions catch real changes. With only `transform`, the table was a tautology.

## Build order

Each step keeps `npm run typecheck && npm run lint && npm run test && npm run dev` green / runnable.

1. **Extend `FieldKind` with `bool` and `enum`.** Update `core/types.ts`. Existing schemas don't reference the new kinds; tests stay green.

2. **Write `collider.ts`.** Six fields per Engine.md §5.3:
   - `offset_x`, `offset_y` — float, default 0
   - `width`, `height` — float, default 32
   - `is_solid` — bool, default true (Engine.md §10 recommendation)

   `serialize` returns `{ offset_x, offset_y, width, height, flags: data.is_solid ? 1 : 0 }`. `deserialize` reads `flags` as uint32; `is_solid = (flags & 1) === 1`. Other bits dropped on read (Engine.md §5.3 reserves them).

   `requires: ['transform']` (Engine.md §13).

   Tests: round-trip for several `is_solid` and bounds combinations; deserialize rejects non-objects and non-numeric fields with clear messages.

3. **Write `physics.ts`.** Five fields per §5.4:
   - `vx`, `vy` — float, default 0
   - `gravity_magnitude` — float, default 0.5 (the Engine.md §9 "feels right" value)
   - `gravity_direction` — enum, default `0` (down). Options: `{ value: 0, label: 'Down' }`, `1: 'Up'`, `2: 'Left'`, `3: 'Right'`.

   `requires: ['transform']`.

   Test the enum round-trip explicitly: `serialize({ ...rest, gravity_direction: 2 })` → `gravity_direction: 2`; `deserialize({ ..., gravity_direction: 2 })` → `2`. Reject out-of-range values with a clear error.

4. **Write `player_controlled.ts`.** No fields, `isFlag: true`, `requires: ['transform', 'physics']`. `serialize` returns `{}` (an empty record — `sceneToJSON` reads `isFlag` and won't write it under `components` anyway, but the shape stays consistent for the registry contract). `deserialize({})` returns `{}`.

   Test: schema metadata, requires array, shape.

5. **Update `schemas/registry.ts`** to include all four schemas. The `Record<string, ComponentSchema<unknown>>` shape doesn't change.

6. **Add `schemas/registry.drift.test.ts`.** A literal table, hand-typed against Engine.md §5:

   ```ts
   const expected = [
     { key: 'transform',         maskBit: 1, sizeBytes: 16, fields: ['x', 'y', 'width', 'height'] },
     { key: 'collider',          maskBit: 3, sizeBytes: 20, fields: ['offset_x', 'offset_y', 'width', 'height', 'is_solid'] },
     { key: 'physics',           maskBit: 4, sizeBytes: 16, fields: ['vx', 'vy', 'gravity_magnitude', 'gravity_direction'] },
     { key: 'player_controlled', maskBit: 5, sizeBytes: 0,  fields: [], isFlag: true },
   ]
   ```

   For each row, assert `REGISTRY[key]` exists and `(maskBit, sizeBytes, fields.map(f => f.name), isFlag ?? false)` matches. The test is deliberately a hand-typed wall — it exists to fail when someone changes `transform.maskBit` to `99` thinking it's harmless. The pain is the point.

   Note: `is_solid` is the *editor's* field name (a bool), but Engine.md §5.3 lists `flags` as the binary field. The drift table uses the editor's field names; the binary side is enforced by the round-trip test in `collider.test.ts` matching the JSON shape `{ flags: 1 }`. Document this in the test file: "drift test pins the schema layer; round-trip tests pin the JSON layer; together they pin both ends."

7. **Add `EditorCore.addComponent` and `removeComponent`.** Both wrapped in an internal transaction so undo collapses them to one step:

   ```ts
   addComponent(id, componentKey) {
     this.beginTransaction()
     try {
       const schema = REGISTRY[componentKey]
       if (!schema) throw new Error(...)
       // assert requires are present
       const entity = this.findEntity(id)
       for (const req of schema.requires ?? []) {
         if (!(req in entity.components)) throw new Error(`requires '${req}'`)
       }
       // assign default
       const components = { ...entity.components, [componentKey]: makeDefault(schema) }
       this.replaceEntity(id, { ...entity, components })
       this.commitTransaction()
     } catch (err) {
       this.abortTransaction()
       throw err
     }
   }
   ```

   `removeComponent` symmetric. Reject if any *other* schema in the entity declares this key in its `requires`. Same try/abort pattern. The mutation itself is one `replaceEntity`, so without the transaction wrapper undo would still work — the wrapper exists to keep the contract uniform: every public `EditorCore` mutation that does work emits exactly one history entry.

   Tests:
   - `addComponent` rejects unknown keys.
   - `addComponent` rejects when `requires` missing.
   - `removeComponent` rejects when another component depends on it.
   - Both go through one history entry; undo restores prior entity state.

8. **Lift `isFlag` schemas in `sceneToJSON`.** In the per-entity loop:

   ```ts
   const components: Record<string, unknown> = {}
   const flags: Record<string, true> = {}
   for (const [key, data] of Object.entries(entity.components)) {
     const schema = REGISTRY[key]
     if (!schema) throw new Error(...)
     if (schema.isFlag) flags[key] = true
     else components[key] = schema.serialize(data)
   }
   return { id: entity.id, ...flags, components }
   ```

   Test: a scene with one entity that has `transform`, `physics`, and `player_controlled` serializes to JSON with `player_controlled: true` at top level and only `transform`/`physics` under `components`. The exact Engine.md §6 hero shape (modulo no `display_name`/`sprite` yet).

9. **PropertyPanel: add bool and enum field branches.** In `FieldInput`:

   ```tsx
   if (field.kind.kind === 'bool') {
     return <label>... <input type="checkbox" checked={!!value} onChange={e => onFieldChange(field.name, e.currentTarget.checked)} /></label>
   }
   if (field.kind.kind === 'enum') {
     return <label>... <select value={value} onChange={e => onFieldChange(field.name, Number(e.currentTarget.value))}>
       {field.kind.options.map(o => <option key={o.value} value={o.value}>{o.label}</option>)}
     </select></label>
   }
   ```

   The transaction wrap from V2.0 still applies — focus on the input opens the transaction; blur closes it. Checkbox toggles fire focus → change → blur in immediate succession (browser behaviour); the transaction collapses to one history entry per click. Same for `<select>` — focus, change selection, blur — one entry.

   Don't reuse the V1.5 number-text-state machinery for these. Bool and enum have no in-progress half-states.

10. **PropertyPanel: add-component dropdown + per-section remove button.** Below the last section:

    ```tsx
    const eligible = Object.values(REGISTRY).filter(s => !(s.key in entity.components))
    return <select onChange={...}>
      <option value="">Add component…</option>
      {eligible.map(s => <option key={s.key} value={s.key} disabled={!requirementsMet(s, entity)}>{s.label}</option>)}
    </select>
    ```

    On change, call `core.addComponent(entity.id, key)`. Errors surface as a transient inline message; don't `alert()`. (V2.3 will replace this with proper validation UI.)

    Per-section remove button: `<button onClick={() => core.removeComponent(entity.id, key)} disabled={isRequiredByPeer(...)}>Remove</button>`. The `disabled` rule is computed from the registry (does any other component on this entity declare this key in its `requires`?); same logic the core enforces. Disabling at the UI level prevents the throw path from ever firing in normal use.

11. **Toolbar entity presets.** `App.tsx`'s "Add Entity" button becomes a `<select>` (or a tiny popover — V2.1 keeps the `<select>` for simplicity). Three presets:
    - **Empty** — `core.addEntity([])`. No components. Entity is unrenderable, exists for UX gymnastics.
    - **Static Solid** — `addEntity(['transform', 'collider'])`. Default Floor-shaped entity.
    - **Dynamic** — `addEntity(['transform', 'physics', 'collider'])`. Default Hero shape minus the player flag.

    Presets live in a small `EntityPresets.tsx` (new file) that exports the array and renders the dropdown; no logic in `App.tsx`. `core.addEntity(componentKeys)` already exists from V1.1 — no core change.

Steps 1–8 are pure logic. Steps 9–11 are UI-only.

## Acceptance criteria

- `npm run typecheck` — clean.
- `npm run lint` — clean.
- `npm run test` — V2.0 specs still green; new schema tests, drift test, addComponent/removeComponent specs all green.
- Drift test sanity check: locally edit `transform.maskBit` from `1` to `99`; rerun `npm run test`; observe drift test fails with a message naming the offending key. Revert.
- `npm run dev` — manual checks:
  - Toolbar "Add Entity" → "Dynamic" → entity appears in canvas as a 32×32 rect at (0,0); panel shows three sections (Transform, Physics, Collider) with their default values rendered correctly (number, enum dropdown, checkbox).
  - In the panel, change `Physics > Gravity direction` from "Down" to "Up" → no visible canvas change (engine has no preview), but the sceneToJSON dump (Show JSON) reflects `gravity_direction: 1`.
  - Toggle `Collider > Solid` off → the JSON for that entity shows `flags: 0`; toggle back on → `flags: 1`.
  - Click "Add Component" on an entity that only has `transform` → dropdown lists every registered component except `transform`. Pick `player_controlled` → it's disabled because `requires: physics` and `physics` isn't there. Add `physics` first, then `player_controlled` → it now adds; toolbar JSON shows `player_controlled: true` at top level (NOT under `components`).
  - Try to remove `transform` from an entity that has `collider` (which requires it) → button is disabled, ARIA tooltip explains why (or for V2.1 a `title` attribute is fine; V2.3 owns proper inline messaging).
  - Press Cmd+Z after an `addComponent` → component is removed in one step. Cmd+Shift+Z → restored. Same for `removeComponent`.

## Open questions to resolve during V2.1

- **Should presets live in `core/` or `ui/`?** Decision: **`ui/`**. Presets are UX (default starting point for a user gesture); they aren't part of the data contract. Same reason `EditorCore` doesn't ship a "default scene" — composition root decides what to add.
- **Should `addComponent` accept a partial override of defaults?** Decision: **no in V2.1**. Adding overrides invents a UX (where do they come from?) for a problem the user can solve in two clicks (add component, edit field). Revisit if duplicate-entity in V3 needs it.
- **Display of `flags` field in the JSON.** Engine.md §5.3 documents `flags` as a `uint32`. The editor's schema field is `is_solid` (bool). This is fine — Engine.md is the runtime contract; the editor is allowed an internal representation as long as `serialize` produces the right shape, which the round-trip test pins.
- **Enum option labels are English.** Localization is not on the V2 horizon; if it ever is, labels become i18n keys and `options[i].label` becomes a function. V2.1 hardcodes.
- **Empty preset.** Decision: keep it. Useful for entities that are pure audio (no transform), once V2.2 lands. Discoverability is fine — it's just one option in a dropdown of three.
- **The drift test and `is_solid`.** Per step 6 above, the drift test checks editor field names, not engine field names. The collider's serialize-to-`flags` is verified by the round-trip test inside `collider.test.ts`. This split is intentional and documented in the drift test header comment.

## What V2.2 unlocks (the next slice after V2.1)

- The schema-driven panel renders `asset-ref` and `colour-abgr` field kinds the same way it now renders `bool` and `enum` — just two more branches in `FieldInput`.
- `AssetStore` interface lands as a new `core/`-adjacent abstraction with a browser implementation in `io/adapters/`. The hexagonal boundary forbids `core/` from importing `io/`, so the AssetStore is injected into `EditorCore` at construction the same way registry already is.
- `sprite` and `audio` schemas reuse the registry pattern from V2.1 directly; their field kinds plug into the same panel.
- Texture rendering on the canvas re-uses the existing `<rect>` placement; the texture is just an `<image>` overlay with a blob URL from the AssetStore. Selection styling unchanged.

V2.2 introduces async I/O — that's the new architectural surface. Everything else is the V2.1 pattern repeated.
