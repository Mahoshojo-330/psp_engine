# V1.1 — Vertical Slice: Core + Schema-Driven Panel (No Canvas, No JSON)

## Goal

Build the **smallest possible end-to-end slice that still tests the load-bearing architecture**: a non-React `EditorCore` owning a `Scene`, React subscribing to it via `useSyncExternalStore`, and a property panel **generated from a schema** rather than hand-coded.

By the end of V1.1, you should be able to:

1. Click an "Add Entity" button.
2. See a new entity appear in a list (`<ul>`), labelled by id.
3. Click an entity in the list — it becomes the selected one.
4. The property panel shows one labelled `<input type="number">` per field of the selected entity's `transform` component, **automatically generated from `transform`'s `FieldSchema[]`**.
5. Edit a field — the value updates in `EditorCore`, both the panel and the list re-render from the new state.

Everything is in-memory. No SVG canvas, no drag, no JSON output, no save/load, no other components.

---

## Why this slice (and what was cut from V1)

V1 (the previous draft) tried to test five new things at once: schema-driven UI, Core ↔ React subscription, the hexagonal boundary, SVG drag-and-drop, and JSON serialization. For a first React project that's too many unknowns failing simultaneously.

V1.1 keeps the **three architectural questions that can only be answered by writing UI**:

- **Schema-driven UI works.** The panel reads `transform.fields` and renders inputs. If this works for `int`/`float`, the pattern extends mechanically to other field kinds in V2.
- **Core ↔ React subscription works.** `EditorCore` owns the scene, mutations from the panel are visible to the entity list and vice versa, with no React state holding scene data.
- **The hexagonal boundary holds.** `core/` and `schemas/` typecheck and lint without importing from `ui/` — the ESLint rule we already wired up actually fires when violated.

V1.1 **defers** the things that are either tested better by unit tests or aren't load-bearing yet:

- **SVG canvas + drag** — moved to V1.5. The property panel already exercises the edit path; SVG just adds pointer-event coordinate math.
- **JSON serialization (`sceneToJSON`)** — moved to V1.5. Serialization shape can be answered by a unit test on `transform.serialize()` once that function exists; it doesn't need a UI button to be validated.
- **Other components and field kinds** — moved to V2. They're mechanical extensions, not architectural questions.

## Build order

Each step is independently runnable — `npm run typecheck` + `npm run lint` should be clean after every step. Tests for steps 2 and 3 are written alongside, not after.

1. **`core/types.ts`** — type defs only, no runtime code. Confirms the shape everything else imports.
2. **`core/EditorCore.ts` + test** — pure TS, no React. Methods: `addEntity(componentKeys)`, `setField(id, componentKey, fieldName, value)`, `selectEntity(id | null)`, `getSnapshot()`, `subscribe(listener)`. Test: add entity → snapshot reflects it; setField → snapshot updates and subscribers fire; selectEntity persists in snapshot.
3. **`schemas/transform.ts` + test** — declarative object matching `ComponentSchema<Transform>`. `maskBit: 1`, four fields with labels and defaults. `serialize`/`deserialize` can be stubbed (`throw new Error('not in V1.1')`) — they're not exercised yet. `validate` returns `[]`. Test: schema metadata matches Engine.md §5.1 (key, maskBit, sizeBytes, ordered field names).
4. **`ui/hooks/useScene.ts`** — `useSyncExternalStore(core.subscribe, core.getSnapshot)`. Three lines.
5. **`ui/EntityList.tsx`** — given `core`, calls `useScene()`, renders `<ul>` of entities. Each row shows `#{id}` (display-name editing is deferred). Clicking a row calls `core.selectEntity(id)`. Selected row is marked with `aria-selected="true"` (styled via `[aria-selected="true"]` in CSS) — communicates selection to assistive tech without inventing a bespoke class.
6. **`ui/PropertyPanel.tsx`** — given `core`, reads selected entity from snapshot. If none selected, shows "no selection" placeholder. Otherwise, looks up the `transform` schema, iterates `schema.fields`, renders one labelled `<input type="number">` per field. `onChange` parses the value and calls `core.setField`.
7. **`App.tsx`** — top toolbar with one button ("Add Entity" → `core.addEntity(['transform'])`). Two-column layout below: `<EntityList core={core}>` on the left, `<PropertyPanel core={core}>` on the right.
8. **`main.tsx`** — `const core = new EditorCore(); root.render(<App core={core} />)`.

## Acceptance criteria

- `npm run typecheck` — clean.
- `npm run lint` — clean. (Confirms boundary rules don't fire on `core/` or `schemas/`.)
- `npm run test` — both unit tests green.
- `npm run dev` — manual checks:
  - Click Add Entity → "#1" appears in the list.
  - Click "#1" in the list → it gets `aria-selected="true"` (visibly distinguished), panel shows four inputs labelled X/Y/Width/Height with defaults.
  - Edit X to `100` → the input keeps the value (state survives re-render); panel re-renders without losing focus catastrophically.
  - Click Add Entity again → "#2" appears, "#1" stays selected, panel still shows #1's values.
  - Click "#2" → panel switches to #2's values.

If any of those four manual checks fails, the architecture has a bug — fix it before moving to V1.5.

## The one advanced React pattern you're learning upfront

`useSyncExternalStore` is React's official escape hatch for "I have a non-React object that holds state, please subscribe to it." It exists because the alternative — putting all state inside React via `useState`/`useReducer` — couples your data layer to the UI framework, which is exactly what the hexagonal boundary forbids.

You're learning it on day one because the architecture depends on it. Everything else in V1.1 (`<ul>`, `<input>`, `onChange`, props) is plain React. The Core class is plain TypeScript. The hook is the seam between them, and it's three lines.

If `useSyncExternalStore` proves painful, the fallback is a thin React Context wrapper around the same `EditorCore` — the Core class itself doesn't change. Don't refactor preemptively; only swap if the actual usage hurts.

## Open questions to resolve during V1.1

- **Selection model.** Stored as `selectedEntityId: EntityId | null` on `EditorCore`. This keeps "what's selected" testable without a browser and means `EntityList` and `PropertyPanel` agree without prop-drilling. Decided up-front rather than left open — the alternative (React state in `App`) was rejected because it splits the source of truth.
- **Default values for new entities.** Live on each `FieldSchema` as a `default`. `addEntity(['transform'])` asks the registry (just `transform.ts` in V1.1) to materialize a default component instance. Decided.
- **EntityId type.** Monotonically incrementing `number` from a counter inside `EditorCore`. Revisit when V2 load-from-disk lands and ID collisions across sessions become possible — branded type or UUID at that point.
- **Component registry shape.** V1.1 has only one schema; importing it directly is fine. The "registry" (a `Map<string, ComponentSchema<unknown>>` keyed by component key) only earns its keep with ≥2 schemas. Decide in V2 when adding the second one.

## What V1.5 unlocks (the next slice after V1.1)

- `<Canvas>` SVG renderer — `viewBox="0 0 480 272"`, one `<rect>` per entity with a transform. Click a rect → `core.selectEntity(id)`. Initially read-only (no drag).
- Pointer-event drag on rects, updating `x`/`y` via `core.setField`. Coordinate translation handled in one place (`screenToCanvas` helper).
- `io/serialization/sceneToJSON.ts` + a "Show JSON" toolbar button that opens a `<textarea>`. Unit test asserts the output for a transform-only entity matches `Engine.md §6` exactly.

V1.5 is small because V1.1 already paid for the hard architectural questions. After V1.5 the editor is genuinely usable for one component, and V2 is mostly "add more files of the same shape".
